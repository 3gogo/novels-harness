# 网络小说多 Agent 生产 Harness 设计

日期：2026-04-07

## 1. 背景与目标

本项目要构建一个适用于本地单人总控的网络小说生产 harness。系统的核心目标不是充当“写作软件”，而是成为一个面向小说试产与筛选的生产编排系统。

用户定位：

- 单人操盘
- 本地运行
- 通过 Web 工作台总控
- 以高自动化批处理为主
- 执行器模型无关，可替换不同模型与平台

业务优先级：

1. 爆款试产
2. 全链路可审计
3. 风格控制
4. 稳定连载

试产通过标准采用双闸门：

- 产出可读的前 3 到 5 章开篇样稿
- 同时产出完整评分、风险与继续/放弃建议

系统主要面向“先批量孵化，再把优胜项目切入长篇线”的混合模式。

## 2. 设计原则

- 编排优先：系统首先解决任务拆解、批处理、晋级、回滚与审计，而不是先做写作编辑器。
- 资产落盘：所有核心小说资产必须保留在文件系统中，不被数据库或 UI 黑盒锁死。
- 模型无关：工作流节点、角色定义和执行器适配彻底解耦。
- 人工闸门最少化：除关键裁决点外，其余流程默认支持无人值守批处理。
- 可晋级可淘汰：系统要擅长快速筛选弱项目，而不是机械地把所有项目都推进到后续阶段。
- 连载延后：长篇稳定生产能力属于优胜项目的第二阶段能力，不进入首版核心。

## 3. 非目标

以下内容不进入 MVP：

- 浏览器内的完整正文编辑器
- 复杂多人协作权限系统
- 自动发布到小说平台
- 100 章以上长篇的全自动连载
- 高复杂度的 agent 社会模拟

## 4. 总体架构

系统采用“三层编排 + 一层资产”的结构。

### 4.1 控制层：Local Control Room

本地 Web 工作台，负责：

- 创建批次
- 设置赛道、约束和评分规则
- 查看项目状态与队列
- 对比候选项目与评分卡
- 在关键节点批准晋级、返工或淘汰

该层是操盘界面，不承担核心资产存储责任。

### 4.2 编排层：Orchestrator Kernel

负责任务编排与流程控制，核心职责包括：

- 工作流状态机
- 节点依赖管理
- 任务拆解与批次展开
- 重试、恢复、回滚
- 自动闸门裁决
- 人工闸门挂起与恢复

该层是整个 harness 的生产内核。

### 4.3 执行层：Execution Mesh

由多个角色化 agent 组成，但这些 agent 不直接依赖某个固定模型。执行层通过适配器调用底层执行器。

它负责：

- 消化输入上下文
- 按角色职责产出结构化结果
- 返回资产、指标和运行轨迹

### 4.4 资产层：File-Backed Asset Layer

所有可长期复用与复盘的核心资产都落到文件系统中，包括：

- 批次简报
- 题材候选
- 概念包
- 开篇样稿
- 评分卡
- 晋级/淘汰决策
- 运行轨迹
- 提示词与模型使用记录

资产层独立于数据库存在，确保系统可审计、可迁移、可回滚。

## 5. 角色团队设计

首版编辑部角色按三组组织。

### 5.1 发现组

负责从批次目标扩散出足够多的候选项目。

包含角色：

- 题材侦察员：根据目标赛道和读者偏好生成候选方向
- 套路混编师：组合套路、反套路和卖点结构
- 定位编辑：产出标题、卖点、对标与差异化定位

### 5.2 生产组

负责把优胜概念转成更接近市场验证的可读资产。

包含角色：

- 概念打包师：补全简介、角色卡、世界设定、章节承诺
- 开篇作者：生成前 3 到 5 章开篇正文
- 钩子强化师：对开篇钩子、节奏和爽点密度做定向补强

### 5.3 评审组

负责严格打分和做去留判断。

包含角色：

- 市场评审：评估卖点、可点击性、连载潜力
- 故事批评师：评估叙事抓力、行文质量、角色驱动
- 晋级裁决器：汇总多项评分，输出 promote / retry / kill 建议

说明：

- 首版虽然存在多个角色，但它们不要求同时并发到极细粒度。
- 在实现上，允许将多个角色合并到同一执行模板中，只要输出边界保持稳定。

## 6. 核心工作流

系统默认围绕批处理孵化线运行。

### 6.1 主流水线

1. `batch_brief`
   输入批次目标、赛道、受众、限制、理想爆点。
2. `idea_spread`
   扩散生成 10 到 30 个候选项目。
3. `concept_pack`
   为优胜候选补全简介、角色、设定、承诺。
4. `opening_draft`
   为少量决赛项目生成前 3 到 5 章。
5. `opening_review`
   多角色评估样稿与概念包。
6. `promotion_decision`
   自动或人工决定晋级、返工、重试或淘汰。

### 6.2 人工闸门

系统仅保留两个高价值人工闸门：

- 闸门 1：概念包排序后的短名单修正
- 闸门 2：开篇样稿与评分卡完成后的最终晋级批准

其余节点默认允许无人值守。

### 6.3 晋级原则

项目满足以下条件时优先晋级：

- 前 3 到 5 章具备稳定抓力
- 核心 premise 适合长线放大
- 主要问题可通过局部修订解决

项目满足以下条件时优先淘汰或重试：

- premise 强但正文落地塌陷
- 评审持续指出钩子密度低、节奏弱、情绪驱动不足
- 只有在重做核心 premise 后才成立

## 7. 文件资产结构

推荐目录结构如下：

```text
workspace/
  batches/
    2026-04-xianxia-hook-batch/
      brief.yaml
      ranking.json
      decisions.json
  projects/
    project-001/
      project.yaml
      bible/
        premise.md
        world.md
        roles.md
        style.md
      incubation/
        idea-cards/
        concept-pack/
        opening-draft/
        review-scorecards/
      serial/
        volume-plan/
        chapter-queue/
        drafted/
        revised/
      lineage/
        runs.jsonl
        prompts/
        model-usage.json
      decisions/
        promote.md
        retry.md
        archive.md
```

设计说明：

- `batches/` 保存批处理运行与排序结果
- `projects/` 保存单书级资产
- `bible/` 是长期稳定上下文，是后续所有 agent 的主输入源
- `incubation/` 只保留试产阶段产物
- `serial/` 为晋级后的连载阶段预留
- `lineage/` 保存运行链路、提示词、模型记录
- `decisions/` 保存可读的决策说明

## 8. Web 工作台模块

首版工作台建议包含 6 个模块。

### 8.1 批次控制台

- 创建和配置批次
- 选择目标赛道与约束
- 发起批跑
- 查看各阶段漏斗数据

### 8.2 项目看板

- 查看每个项目当前阶段
- 查看失败原因、待审批项和最近产物
- 查看项目是否处于晋级、返工、归档状态

### 8.3 评审台

- 对比多个题材卡
- 对比多个概念包
- 对比多个开篇样稿
- 对比多个评分卡并做晋级决策

### 8.4 资产阅读器

- 阅读简介、大纲、样稿、角色卡、世界观
- 查看历史版本与关联决策

### 8.5 规则中心

- 配置角色模板
- 配置评分 rubric
- 配置晋级规则
- 配置禁用套路与风格约束

### 8.6 运行审计台

- 查看每次运行使用的模型与角色
- 查看消耗、失败点、重试情况
- 查看是否允许重跑与推荐修复动作

工作台的重点是“批量对比与决策”，而不是“浏览器内长文编辑”。

## 9. 模型无关执行器抽象

系统需要彻底拆开“工作流节点”“角色职责”“底层模型调用”。

### 9.1 Workflow Node

定义某一阶段的业务责任：

- 输入资产
- 输出资产
- 验收标准
- 失败分型
- 是否允许自动重跑

例如：

- `idea_spread`
- `concept_pack`
- `opening_draft`
- `opening_review`
- `promotion_decision`

### 9.2 Role Profile

定义角色职责与输出风格，不直接绑定模型。

例如：

- 开篇作者
- 市场评审
- 晋级裁决器

Role Profile 需要至少包含：

- 角色名称
- 职责说明
- 允许使用的上下文范围
- 目标输出格式
- 评分或反馈格式

### 9.3 Executor Adapter

负责对接具体执行器。

适配器职责：

- 接收统一任务合同
- 调用目标平台或模型
- 归一化输出
- 返回状态、资产和追踪信息

首版至少支持：

- 第一种常用执行器
- 第二种用于证明接口可切换的执行器

### 9.4 Run Contract

统一执行协议建议包含：

- `context_refs`
- `task_brief`
- `constraints`
- `output_schema`
- `artifact`
- `metrics`
- `status`
- `trace`

这样工作流无需知道底层模型细节，只消费标准化结果。

## 10. 失败恢复与重跑策略

首版必须内建失败分型，否则高自动化批处理不可维护。

### 10.1 失败分类

- `retryable`
  临时问题，可自动重试。例如超时、模型拒答、格式偏差。
- `repairable`
  产物不合格，但可由修复节点补救。例如钩子不足、结构不达标。
- `review_required`
  系统不能安全自动裁决，需人工介入。
- `terminal`
  本轮项目直接淘汰，无继续投入价值。

### 10.2 节点级记录

每个节点执行都应保存：

- 输入版本
- 输出版本
- 使用角色
- 使用执行器与模型
- 产物路径
- 指标与评分
- 失败分类
- 重跑次数
- 是否允许恢复

### 10.3 恢复动作

系统至少支持：

- 原节点重跑
- 从指定节点恢复
- 修复节点补跑后回接主线
- 直接归档失败项目

## 11. MVP 范围

首版只做足够支撑试产和筛选的能力。

### 11.1 包含

1. 批次创建与项目队列
2. `idea -> concept_pack -> opening_draft -> review -> promote/kill` 主流水线
3. 角色配置与评分规则配置
4. 文件资产落盘与 lineage 追踪
5. 本地 Web 工作台的批次页、项目页、评审页
6. 至少两类执行器适配接口

### 11.2 不包含

- 长篇大规模自动连载
- 平台发布能力
- 多人协作权限
- 浏览器正文编辑器
- 高复杂度 agent 群体自治

## 12. 推荐实施分期

### Phase 1：编排核心与资产协议

- 定义项目结构与文件协议
- 定义工作流状态机
- 定义节点合同与角色模板
- 打通最小执行器适配链路

### Phase 2：试产主流水线

- 实现批次简报到晋级裁决的完整流水线
- 落地评分卡和失败分型
- 支持批量发起、失败重跑与项目归档

### Phase 3：本地 Web 工作台

- 批次控制台
- 项目看板
- 评审台
- 资产阅读器

### Phase 4：长篇线与风格控制增强

- 胜出项目切换到连载模式
- 强化 bible 驱动的一致性约束
- 引入风格巡检与设定冲突检查

## 13. 风险与注意点

- 如果过早把角色拆得过细，首版会增加编排负担；实现时应优先保证协议稳定，再决定并发粒度。
- 如果不把评分卡做成结构化结果，后续晋级与复盘会退化成人工阅读堆积。
- 如果资产只存数据库，后续迁移模型或改 UI 的成本会明显升高。
- 如果工作台变成正文编辑器，会稀释系统的核心价值。

## 14. 结论

该 harness 的推荐路线是：

- 采用工作流引擎型架构作为主干
- 用文件资产层承担审计与持久化
- 用角色化 agent 提供编辑部式协作
- 用模型无关执行器抽象保证后续切换能力

首版只聚焦“批量试产、开篇验证、评分裁决、晋级筛选”，等优胜项目形成稳定漏斗后，再扩展到长篇连载生产。
